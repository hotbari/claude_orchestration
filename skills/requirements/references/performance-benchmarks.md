# 서비스 유형별 성능 타겟

## 개요

이 문서는 웹 서비스 유형별 성능, 가용성, 확장성 벤치마크를 정리합니다. Spec-Writer 에이전트가 Stage 6 (비기능 요구사항)에서 성능 타겟을 제시할 때 참조합니다.

---

## 1. Web Vitals (Google Core Web Vitals)

### 기준값

| 지표 | 양호 (Good) | 개선 필요 | 나쁨 (Poor) | 설명 |
|------|------------|-----------|------------|------|
| LCP (Largest Contentful Paint) | ≤ 2.5s | ≤ 4.0s | > 4.0s | 주요 콘텐츠 로드 시간 |
| INP (Interaction to Next Paint) | ≤ 200ms | ≤ 500ms | > 500ms | 인터랙션 응답성 |
| CLS (Cumulative Layout Shift) | ≤ 0.1 | ≤ 0.25 | > 0.25 | 레이아웃 안정성 |
| FCP (First Contentful Paint) | ≤ 1.8s | ≤ 3.0s | > 3.0s | 첫 콘텐츠 표시 |
| TTFB (Time to First Byte) | ≤ 800ms | ≤ 1.8s | > 1.8s | 서버 응답 시간 |

### 서비스 유형별 LCP 타겟

| 서비스 유형 | LCP 타겟 | 근거 |
|------------|---------|------|
| 랜딩 페이지 | ≤ 1.5s | 첫인상, 전환율 |
| 블로그/콘텐츠 | ≤ 2.0s | 읽기 경험 |
| 이커머스 | ≤ 2.0s | 구매 전환율 |
| 대시보드 | ≤ 2.5s | 인증 후 로드 |
| SaaS 앱 | ≤ 3.0s | 복잡한 UI |

**참고:** Google 연구에 따르면 LCP가 1초에서 3초로 증가하면 이탈률이 32% 증가합니다.

## 3. 서비스 유형별 규모 타겟

### 소규모 (Small)

```
동시 사용자: < 100
일일 요청: < 10,000
데이터: < 1GB
```

**인프라:**
- 단일 서버 (2 vCPU, 4GB RAM)
- PostgreSQL 단일 인스턴스
- 캐싱: 불필요 또는 인메모리

### 중규모 (Medium)

```
동시 사용자: 100 ~ 10,000
일일 요청: 10,000 ~ 1,000,000
데이터: 1GB ~ 100GB
```

**인프라:**
- 앱 서버 2-4대 + 로드밸런서
- PostgreSQL Primary + Read Replica
- Redis 캐싱 (세션, 빈번한 쿼리)
- CDN (정적 자산)

### 대규모 (Large)

```
동시 사용자: 10,000+
일일 요청: 1,000,000+
데이터: 100GB+
```

**인프라:**
- 오토스케일링 앱 서버
- PostgreSQL 클러스터 (파티셔닝, 샤딩)
- Redis Cluster
- CDN + Edge Computing
- 메시지 큐 (비동기 처리)

## 7. 실시간 요구사항

### WebSocket vs SSE vs Polling

| 기준 | WebSocket | SSE | Long Polling |
|------|-----------|-----|-------------|
| 방향 | 양방향 | 서버→클라이언트 | 클라이언트→서버 |
| 연결 수 | 서버당 ~50K | 서버당 ~100K | 요청마다 새 연결 |
| 브라우저 지원 | 모두 | 모두 (IE 제외) | 모두 |
| 프록시 친화 | 낮음 | 높음 | 높음 |
| 적합 사례 | 채팅, 게임 | 알림, 피드 | 간단한 업데이트 |

### 실시간이 필요한 기능

| 기능 | 추천 | 근거 |
|------|------|------|
| 채팅 | WebSocket | 양방향, 낮은 지연 |
| 알림 | SSE | 서버→클라이언트만 |
| 대시보드 업데이트 | SSE 또는 Polling | 낮은 빈도 |
| 협업 편집 | WebSocket + CRDT | 양방향, 충돌 해결 |
| 실시간 검색 | Debounced HTTP | 충분히 빠름 |
