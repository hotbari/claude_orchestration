# 인증 패턴 카탈로그

## 개요

이 문서는 웹 서비스에서 사용되는 주요 인증 패턴을 비교합니다. Spec-Writer 에이전트가 Stage 4 (API 설계)에서 인증 전략을 추천할 때 참조합니다.

---

## 1. JWT (JSON Web Token)

### 작동 방식
- 로그인 시 서버가 서명된 토큰(Access + Refresh)을 발급
- 클라이언트가 `Authorization: Bearer {token}` 헤더로 요청
- 서버는 토큰 서명을 검증하여 인증 (DB 조회 불필요)

### 토큰 구조
```
Header.Payload.Signature
```
- **Access Token**: 짧은 수명 (15분~1시간), 요청 인증용
- **Refresh Token**: 긴 수명 (7일~30일), Access Token 갱신용

### 장점
- **스테이트리스**: 서버 메모리/DB에 세션 저장 불필요
- **수평 확장 용이**: 로드밸런서 뒤 여러 서버에서 동일하게 검증
- **마이크로서비스 친화**: 서비스 간 토큰 공유 가능
- **모바일 친화**: 쿠키 없이 헤더로 전송

### 단점
- **즉시 폐기 불가**: 발급된 토큰은 만료 전까지 유효
- **토큰 크기**: 쿠키보다 큼 (1KB+), 매 요청에 전송
- **보안 저장**: 브라우저에서 안전한 저장 위치 필요 (XSS 취약점)

### 보안 요구사항
- Access Token: httpOnly 쿠키 또는 메모리 저장 (localStorage 지양)
- Refresh Token: httpOnly + Secure + SameSite 쿠키
- 토큰 로테이션: Refresh 사용 시 새 Refresh Token 발급
- 블랙리스트: 로그아웃/비밀번호 변경 시 Refresh Token 무효화

### 업계 사례
- **Notion**: JWT + Refresh Token 로테이션
- **Stripe Dashboard**: JWT + 단기 세션
- **GitHub API**: Personal Access Token (JWT 기반)
- **대부분의 모던 SPA**: React/Vue/Angular 앱

### 적합한 서비스
- SPA (Single Page Application)
- 모바일 앱 백엔드
- 마이크로서비스 아키텍처
- API 우선 서비스

## 2. OAuth2 / 소셜 로그인

### 작동 방식
- 사용자가 외부 제공자(Google, GitHub 등)로 인증
- 제공자가 Authorization Code 발급
- 서버가 Code로 Access Token 교환
- 사용자 정보 조회 후 내부 세션/JWT 생성

### 주요 플로우
| 플로우 | 용도 |
|--------|------|
| Authorization Code | 서버사이드 웹 앱 (가장 안전) |
| Authorization Code + PKCE | SPA, 모바일 앱 |
| Client Credentials | 서버 간 통신 |

### 장점
- **비밀번호 관리 불필요**: 사용자 비밀번호를 저장하지 않음
- **높은 전환율**: 소셜 로그인은 가입 장벽 감소 (30-50% 전환율 향상)
- **신뢰도**: 검증된 이메일, 프로필 정보 즉시 확보
- **보안 위임**: 인증 보안을 대형 제공자에게 위임

### 단점
- **외부 의존성**: 제공자 서비스 장애 시 로그인 불가
- **구현 복잡성**: 여러 제공자 지원 시 복잡도 증가
- **개인정보**: 제공자별 동의 범위(scope) 관리 필요
- **오프라인 사용 불가**: 인터넷 연결 필수

### 인기 제공자
| 제공자 | 적합한 서비스 |
|--------|-------------|
| Google | 일반 소비자 앱, 업무 도구 |
| GitHub | 개발자 도구, 기술 커뮤니티 |
| Apple | iOS 앱 (Apple 가이드라인 필수) |
| Kakao/Naver | 한국 서비스 |
| Discord | 커뮤니티, 게임 관련 |

### 업계 사례
- **Medium**: Google/Facebook/Apple 로그인
- **Dev.to**: GitHub/Twitter 로그인
- **Figma**: Google SSO + 이메일
- **Vercel**: GitHub/GitLab/Bitbucket

### 적합한 서비스
- 소비자 대상 서비스 (B2C)
- 개발자 도구 (GitHub OAuth)
- 빠른 온보딩이 중요한 서비스
- 비밀번호 관리 부담을 줄이고 싶은 서비스

## 3. Session 기반 인증

### 작동 방식
- 로그인 시 서버가 세션 ID 생성, 서버 메모리/DB에 저장
- 세션 ID를 `Set-Cookie` 헤더로 클라이언트에 전달
- 브라우저가 자동으로 매 요청에 쿠키 포함
- 서버가 세션 ID로 사용자 정보 조회

### 장점
- **즉시 폐기 가능**: 서버에서 세션 삭제하면 즉시 무효화
- **서버 제어**: 동시 세션 수 제한, 강제 로그아웃 가능
- **작은 전송 크기**: 세션 ID만 전송 (32-64 bytes)
- **브라우저 네이티브**: 쿠키 자동 전송, 별도 클라이언트 코드 불필요

### 단점
- **스테이트풀**: 서버 메모리/DB에 세션 저장 필요
- **수평 확장 어려움**: 세션 공유를 위해 Redis 등 필요
- **CSRF 취약**: 쿠키 자동 전송으로 CSRF 공격 취약
- **모바일 비친화**: 네이티브 앱에서 쿠키 관리 복잡

### 세션 저장소 옵션
| 저장소 | 특성 |
|--------|------|
| 메모리 (기본) | 빠르지만 서버 재시작 시 세션 손실, 단일 서버만 |
| Redis | 빠르고 공유 가능, 프로덕션 권장 |
| PostgreSQL | 느리지만 별도 인프라 불필요 |

### 보안 요구사항
- `httpOnly`: JavaScript에서 쿠키 접근 차단
- `Secure`: HTTPS에서만 쿠키 전송
- `SameSite=Lax/Strict`: CSRF 방어
- CSRF 토큰: 상태 변경 요청에 추가 토큰 검증

### 업계 사례
- **은행 앱**: 세션 기반 + 단기 만료 (5-15분)
- **GitHub Web**: 세션 쿠키 + CSRF 토큰
- **전통적 MPA**: Rails, Django, Express 기본 인증

### 적합한 서비스
- 서버 렌더링 앱 (SSR/MPA)
- 즉시 세션 폐기가 필요한 서비스 (금융, 의료)
- 관리자 대시보드
- 단일 서버 배포

## 4. API Key

### 작동 방식
- 서버가 사용자/앱별 고유 API Key 발급
- 클라이언트가 `X-API-Key` 헤더 또는 쿼리 파라미터로 전송
- 서버가 Key 조회하여 인증 및 권한 확인

### 장점
- **구현 단순**: 문자열 비교만으로 인증
- **서비스 간 통신**: 서버 대 서버 호출에 적합
- **사용량 추적**: Key별 API 사용량 모니터링 용이
- **다중 Key**: 용도별 여러 Key 발급 가능

### 단점
- **보안 약함**: Key 노출 시 전체 접근 가능
- **사용자 인증 부적합**: 브라우저에 Key 노출 위험
- **갱신 어려움**: Key 교체 시 모든 클라이언트 업데이트 필요

### 업계 사례
- **Stripe API**: 비밀 키(서버) + 공개 키(클라이언트)
- **OpenAI API**: Bearer Token 형식 API Key
- **Google Maps**: URL 파라미터 API Key

### 적합한 서비스
- 공개 API (개발자 대상)
- 서버 간 통신
- 서드파티 통합
- 사용량 기반 과금 서비스

## 6. 서비스 유형별 추천

| 서비스 유형 | 1순위 추천 | 2순위 추천 | 근거 |
|------------|-----------|-----------|------|
| SPA + REST API | JWT | OAuth2 + JWT | 스테이트리스, 수평 확장 |
| 소셜 플랫폼 | OAuth2 + JWT | JWT | 빠른 가입, 소셜 연동 |
| 관리자 대시보드 | Session | JWT | 즉시 폐기, RBAC |
| 블로그/CMS | JWT | OAuth2 | 작성자 인증 + 공개 읽기 |
| 이커머스 | JWT + OAuth2 | Session | 게스트 체크아웃 + 회원 |
| 공개 API | API Key + JWT | API Key | 개발자 인증 + 사용량 추적 |
| 내부 도구 | Session | SSO/SAML | 기업 보안 정책 |
| 실시간 앱 | JWT | Session | WebSocket 핸드셰이크 |

## 7. 하이브리드 전략

### JWT + OAuth2 (가장 일반적)
```
사용자 → OAuth2 로그인 → 서버가 JWT 발급 → 이후 JWT로 API 호출
```
- 소셜 로그인의 편의성 + JWT의 스테이트리스 이점
- 대부분의 모던 웹 서비스에 적합

### API Key + JWT (공개 API)
```
개발자 → API Key로 인증 → JWT 발급 → API 호출
```
- API Key로 개발자 식별 + JWT로 세분화된 권한
- Stripe, Twilio 방식

### Session + JWT (마이그레이션)
```
기존 세션 인증 → 점진적으로 JWT 도입 → 최종적으로 JWT 전환
```
- 레거시 시스템에서 모던 인증으로 전환 시
