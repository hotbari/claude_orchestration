# 코드 리뷰 체크리스트

## 차원 1: 스펙 준수 (Spec Compliance)

### 기능 완성도
- [ ] 요구사항 명세서(requirements.md)의 모든 P0 기능이 구현되었는가?
- [ ] 요구사항 명세서의 P1 기능이 구현되었는가?
- [ ] API 명세서(api-spec.md)의 모든 엔드포인트가 구현되었는가?
- [ ] 각 엔드포인트의 요청/응답 스키마가 명세와 일치하는가?

### 데이터 모델
- [ ] 기술 설계서(technical-spec.md)의 데이터 모델이 정확히 구현되었는가?
- [ ] 모든 제약조건(FK, UNIQUE, NOT NULL 등)이 적용되었는가?
- [ ] 인덱스가 설계대로 생성되었는가?

### 에러 처리
- [ ] 명세된 에러 코드가 모두 구현되었는가?
- [ ] 에러 응답 포맷이 명세와 일치하는가?

## 차원 2: 보안 (Security)

### 인증/인가
- [ ] 보호된 엔드포인트에 인증 미들웨어가 적용되었는가?
- [ ] 역할 기반 접근 제어(RBAC)가 올바르게 구현되었는가?
- [ ] 비밀번호가 안전하게 해시 처리되는가? (bcrypt/argon2)
- [ ] JWT 토큰에 적절한 만료 시간이 설정되었는가?

### 입력 검증
- [ ] 모든 API 입력이 Pydantic 모델로 검증되는가?
- [ ] 문자열 필드에 max_length가 설정되었는가?
- [ ] 숫자 필드에 범위 제한이 설정되었는가?
- [ ] 파일 업로드 시 크기/타입 검증이 있는가?

### 취약점
- [ ] SQL Injection 취약점이 없는가? (Raw SQL 사용 금지)
- [ ] XSS 취약점이 없는가? (dangerouslySetInnerHTML 금지)
- [ ] CSRF 보호가 적용되었는가?
- [ ] 민감 정보(비밀번호, 토큰)가 로그/응답에 노출되지 않는가?
- [ ] .env 파일이 .gitignore에 포함되었는가?

## 차원 3: 성능 (Performance)

### 데이터베이스
- [ ] N+1 쿼리 문제가 없는가? (eager loading 사용)
- [ ] 목록 조회 시 페이지네이션이 적용되었는가?
- [ ] 필요한 인덱스가 생성되었는가?
- [ ] 불필요한 데이터 로딩(SELECT *)이 없는가?

### API
- [ ] 응답 데이터 크기가 적절한가?
- [ ] 불필요한 직렬화/역직렬화가 없는가?
- [ ] 비동기 처리가 적절히 활용되었는가? (async/await)

### 프론트엔드
- [ ] 불필요한 리렌더링이 없는가? (React.memo, useMemo)
- [ ] 큰 목록에 가상화(virtualization)가 필요하지 않은가?
- [ ] 이미지/에셋 최적화가 되었는가?

## 차원 4: 테스트 적정성 (Test Quality)

### 커버리지
- [ ] 모든 API 엔드포인트에 테스트가 있는가?
- [ ] 정상 케이스(happy path) 테스트가 있는가?
- [ ] 에러 케이스(error path) 테스트가 있는가?
- [ ] 엣지 케이스 테스트가 있는가? (빈 입력, 최대값, 경계값)

### 테스트 품질
- [ ] 테스트가 독립적으로 실행 가능한가? (격리)
- [ ] 테스트 이름이 명확한가? (Given-When-Then)
- [ ] 하나의 테스트가 하나의 동작만 검증하는가?
- [ ] Mock/Stub 사용이 적절한가?

### 테스트 실행
- [ ] 모든 테스트가 통과하는가? (`pytest` / `vitest run`)
- [ ] 테스트 실행 시간이 합리적인가?

## 차원 5: 코드 품질 (Code Quality)

### 스타일
- [ ] Python: PEP8 / snake_case 준수
- [ ] React: camelCase / PascalCase 준수
- [ ] import 순서 규칙 준수
- [ ] 타입 힌트가 적절히 사용되었는가?

### 구조
- [ ] 관심사 분리 원칙이 지켜졌는가? (Router → Service → Repository)
- [ ] 불필요한 복잡성이 없는가? (KISS)
- [ ] 중복 코드가 없는가? (DRY, 단 과도한 추상화 주의)
- [ ] 함수/메서드 크기가 적절한가? (30줄 이내 권장)

### 에러 핸들링
- [ ] bare except 사용이 없는가?
- [ ] 적절한 HTTP 상태 코드가 반환되는가?
- [ ] 에러 로깅이 적절한가?
- [ ] 사용자에게 친화적인 에러 메시지가 반환되는가?

## 심각도 분류 기준

### 🔴 Critical (즉시 수정 필요)
- 보안 취약점 (SQL Injection, XSS, 인증 우회)
- 데이터 손실/손상 가능성
- 필수 기능(P0) 미구현 또는 오작동
- 테스트 실패

> **Critical이 1건 이라도 있으면 `REVIEW_FAILED`로 판정**

### 🟡 Warning (수정 권장)
- 성능 이슈 (N+1 쿼리, 불필요한 로딩)
- 핵심 경로 테스트 부족
- 코드 스타일 불일치
- 에러 핸들링 부족

### 🟢 Suggestion (개선 제안)
- 더 나은 패턴/라이브러리 제안
- 코드 가독성 개선
- 문서화 추가 제안
- 미래 확장성 고려
